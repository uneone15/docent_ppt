VOICE TUNING

기계 음성 대신,
가이드 페르소나에 맞게 튜닝

유저리서치에서 "도슨트 음성 톤이 좋았으면 좋겠다"는 니즈를 반영하여 가이드에 맞는 톤을 튜닝

1임다미 페르소나 설정 — 20-30대 대상의 친구형 캐릭터. 격식체 대신 "있잖아요", "근데" 같은 구어체를 쓰고, 강의하듯 설명하지 않고 옆에서 같이 걷는 느낌을 주도록 방향을 잡았다.
2강조·쉼 구분 작업 — 캐릭터의 말투에 맞춰 강조해야 할 단어와 쉬어야 할 대목을 스크립트 단위로 구분해 톤을 조정했다. 문장을 그냥 다 읽어주는 것과, 리듬을 살려 들려주는 것은 체감이 다르다.

Insight

같은 문장이라도 기계처럼 또박또박 읽으면 정보가 되고, 사람처럼 억양과 강조와 쉼을 살려 들려주면 이야기가 된다.

DATA PIPELINE

가져온 데이터를 그대로 쓰지 않았다

서울시 공공기관·언론기사·학술논문 등 공신력 있는 출처를 기반으로 수집.
(※ 학습·연구 목적 프로토타입 기준 활용, 상업화 시 라이선스 별도 검토 예정)

스캔 PDF OCR이미지 PDF파일 → pytesseract(Tesseract OCR의 파이썬 래퍼) + pdf2image + PIL 전처리로 텍스트화.
노이즈 정제검색 정확도를 위해 본문과 무관한 노이즈를 LLM 그대로 인용하지 않도록 정제.
청킹문단→줄바꿈→문장 순으로 우선순위를 둬 재귀적으로 분할. RecursiveCharacterTextSplitter(langchain)로 앞뒤 문맥이 끊기는 걸 방지 → 800자 단위(100자 겹침)로 나눠 맥락 손실이 최소화

ANSWER QUALITY TEST

답변을 못하는 챗봇의 검색 로직 재설계

챗봇의 답변 퀄리티를 테스트하는 과정에서 발견된 문제를 코드 레벨에서 원인 진단하고, 검색 로직 자체를 다시 설계했다

문제 발견

질문 입력 시

1"이야기 질문"과 "장소 질문"으로 로직 설계
2질문 키워드가 해당 로직에 잘못 매칭
3대답이 0건으로 처리되는 오류

원인 해결

1질문을 "이야기 질문" "장소 질문" 사전 분류 로직 제거
2테이블을 함께 검색
3테이블 자체는 분리되어있어 섞임 방지까지 처리

Insight

챗봇이 "모른다"고 한 건 정말 몰라서가 아니라, 질문을 분류하는 판단 로직이 틀렸기 때문 → 판단 자체를 프롬프트가 아니라 코드로 다시 설계
